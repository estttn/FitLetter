## Целевая позиция Бюро Скаут

> **Self-hosted конвейер качества:** найти → отфильтровать → письмо → *вы* откликаетесь → учёт.  
> Не конкурируем с Quick Offer по объёму автооткликов; конкурируем по **контролю, приватности и качеству письма**.

---

## Спринт 1 — «HH без лишних кликов» (≈2 недели)

**Цель:** подтянуть UX к [HH Assistant](https://github.com/Morzh777/HHAssistant) / [Сопровод](https://soprovodai.ru/tracker), остаться без автоотклика.

| # | Задача | Конкурент-ориентир | Приоритет | Критерий готовности |
|---|--------|-------------------|-----------|---------------------|
| 1.1 | **Chrome-расширение**: на странице вакансии HH — «Письмо Бюро Скаут» (берёт URL/id, дергает API, вставляет текст в поле) | HH Assistant | P0 | Отклик с письмом за &lt;30 сек без переключения вкладок |
| 1.2 | API-токен / cookie-сессия для расширения (без пароля в расширении) | — | P0 | Авторизация только на своём инстансе |
| 1.3 | Улучшить трекер: фильтр по дате отклика, поиск по компании/названию | Сопровод | P1 | Найти отклик за 5 сек при 100+ записях |
| 1.4 | Бейдж «письмо pending» + автообновление строки после фоновой генерации (SSE или polling уже есть — довести UI) | — | P1 | Не нужен F5 для готового письма |
| 1.5 | Страница **«Аналитика»**: отклики / приглашения / отказы / % по неделям | Сопровод | P2 | Видно, работает ли стратегия поиска |

**Метрика спринта:** время от «увидел вакансию» до «отклик с письмом» ↓ в 2+ раза.

## Спринт 2 — «Умнее отбор» (≈2 недели)

**Цель:** обогнать правила-скорер по полезности отбора ([hh-ru-apply](https://github.com/Steev193/hh-ru-apply), HH Assistant).

| # | Задача | Конкурент-ориентир | Приоритет | Критерий готовности |
|---|--------|-------------------|-----------|---------------------|
| 2.1 | **Красные флаги вакансии** (релокация, стажёр, «английский C1», переработки) — блок в UI + опциональный auto-skip | HH Assistant | P0 | 5–10 правил + показ причины skip |
| 2.2 | LLM-комментарий к fit: «почему 65%, не 90%» (1–2 предложения, дешёвая модель) | hh-ru-apply | P1 | Понятно, откликаться или нет |
| 2.3 | Дедуп вакансий (одинаковые title+company, разные hh_id) | Quick Offer | P2 | Меньше шума в таблице |
| 2.4 | Настройка профиля в UI (queries, salary_min, exclude) без правки JSON | — | P1 | Пользователь меняет фильтры сам |
| 2.5 | Экспорт письма в **буфер / .txt** одной кнопкой (уже есть copy — добавить «скачать») | Genspark/Сопровод | P2 | — |

**Метрика спринта:** доля откликов с последующим «приглашение» ↑ (мерить в 2.5 аналитике).

## Спринт 3 — «Продукт и надёжность» (≈2 недели)

**Цель:** зрелость self-hosted SaaS; не обязательно автоотклик.

| # | Задача | Конкурент-ориентир | Приоритет | Критерий готовности |
|---|--------|-------------------|-----------|---------------------|
| 3.1 | Онбординг: первый вход → загрузить резюме → один тестовый сбор | Сопровод | P0 | Новый user до первой вакансии &lt;10 мин |
| 3.2 | Уведомления: Telegram-бот «N писем готово» / «сбор завершён» | JobMeister | P1 | Не сидеть на странице при 100 письмах |
| 3.3 | Резервный сбор: при 403 HH — понятная ошибка + совет (VPN, cookies, снизить workers) | — | P0 | Не «тихий ноль вакансий» |
| 3.4 | `git pull` на сервере в deploy (опционально) + health с `rev` (✅ уже есть) | — | P1 | Деплой предсказуемый |
| 3.5 | **Опционально:** полуавтоотклик через Playwright (отдельный флаг, default OFF, предупреждение о рисках HH) | Quick Offer, hh-ru-apply | P3 | Только после явного согласия в настройках |

**Метрика спринта:** uptime сбора, меньше ручных деплоев, NPS от 2–3 активных пользователей.
